# 中文语法纠错多候选对齐与高亮展示工具方案（人读版）

## 1. 项目目标

本工具用于展示中文语法纠错任务中，原句、参考答案和多个候选句之间的修改差异。

工具第一版只解决一个核心问题：

> 原句中的哪个片段，在参考答案和不同候选句中分别变成了什么？

本版本只做差异观察，不做自动判错。具体包括：

1. 支持手动输入原句、参考答案和候选句；
2. 支持上传 JSON 文件；
3. 支持单样本 JSON 和多样本 JSON 数组；
4. 以原句为中心进行字符级对齐；
5. 提取 `replace`、`delete`、`insert` 三类基础编辑；
6. 合并相邻小编辑；
7. 合并重叠编辑区间，生成 `edit_groups`；
8. 生成 `render_lines`，用于稳定渲染多行高亮视图；
9. 用编辑组表格展示不同句子对同一原句片段的修改；
10. 用多行高亮视图展示完整句子中的修改位置；
11. 支持删除、插入、替换的可视化；
12. 支持不可见字符可视化；
13. 支持长句自动换行；
14. 支持多样本切换查看；
15. 上传多样本 JSON 后只对当前样本按需计算和渲染。

本版本不做以下内容：

1. 不判断修改是否正确；
2. 不判断过纠；
3. 不判断漏纠；
4. 不判断疑似正确；
5. 不做语义合理性判断；
6. 不做自动评分；
7. 不接入 ChERRANT；
8. 不单独识别调序，调序统一作为替换展示。

## 2. 总体设计原则

### 2.1 以原句为中心对齐

所有参考答案和候选句都统一与原句对齐。

不做：

```text
候选A vs 候选B
候选A vs 参考答案
候选B vs 参考答案
```

而做：

```text
原句 vs 参考答案1
原句 vs 参考答案2
原句 vs 候选句1
原句 vs 候选句2
原句 vs 候选句3
```

这样所有修改都映射到同一个原句坐标系中，便于比较不同句子对同一原句片段的修改方式。

### 2.2 字符级对齐，展示级合并

中文没有天然空格，词级分词可能不稳定，因此第一版使用字符级 diff。

但是字符级 diff 可能太碎，例如：

```text
感到很有兴趣 → 很感兴趣
```

底层可能得到：

```text
删除：到
删除：有
```

展示时需要合并成：

```text
感到很有兴趣 → 很感兴趣
```

因此，本工具分为两层：

1. 底层字符级对齐：用于定位差异；
2. 展示级编辑合并：用于让用户更自然地理解修改。

### 2.3 只展示差异，不判断对错

本版本只展示：

```text
谁在原句哪个位置改成了什么
```

不自动标注：

```text
正确 / 错误 / 过纠 / 漏纠
```

### 2.4 调序统一作为替换展示

本版本不单独识别调序或交换位置。

例如：

```text
原句：我把作业昨天写完了。
候选：我昨天把作业写完了。
```

展示为：

```text
把作业昨天 → 昨天把作业
```

即调序统一作为 `replace` 展示，不生成 `move` 或 `reorder` 类型。

## 3. 输入方式

本版本支持两种输入方式：

```text
方式一：单样本手动输入
方式二：上传 JSON 文件导入
```

两种输入方式最终都会转换成统一样本结构，然后复用同一套对齐、编辑提取、编辑组合并和高亮展示流程。

## 4. 单样本手动输入

### 4.1 手动输入区

手动输入模式下，页面包含：

```text
【原句】
固定一个输入框

【参考答案】
动态输入列表，可增加、删除，可为空

【候选句】
动态输入列表，可增加、删除，至少一条
```

页面示意：

```text
【原句】
[ 他对这个问题感到很有兴趣。 ]

【参考答案】
[ 他对这个问题很感兴趣。 ]      [删除]
[ 他对这个问题很有兴趣。 ]      [删除]
[ 添加参考答案 ]

【候选句】
[ 他对这个问题很感兴趣。 ]      [删除]
[ 他对这个问题非常感兴趣。 ]    [删除]
[ 他对这个题目很感兴趣。 ]      [删除]
[ 添加候选句 ]

[生成对齐结果] [清空]
```

### 4.2 手动输入校验规则

| 输入项 | 规则 |
|---|---|
| 原句 | 必填，只能有一个 |
| 参考答案 | 可为空，可添加多条，可删除 |
| 候选句 | 至少一条，可添加多条，可删除 |
| 空参考答案输入框 | 生成结果时自动忽略 |
| 空候选句输入框 | 生成结果时自动忽略 |
| 参考答案编号 | 按当前顺序生成 `ref_1`、`ref_2` |
| 候选句编号 | 按当前顺序生成 `candidate_1`、`candidate_2` |

如果用户删除中间某条候选句，生成结果时按照剩余候选句的当前顺序重新编号。

### 4.3 手动输入空格保留规则

手动输入时，判断输入框是否为空可以使用 `trim()`，但保存文本时必须保留用户原始输入内容。

也就是说：

```text
如果 value.trim() === ""，则视为空输入框并忽略；
否则保留 value 原文，不自动 trim。
```

原因是中文语法纠错中可能需要观察句首空格、句尾空格或句中空格的删除和插入。如果保存时自动 trim，会导致这些修改无法展示。

例如：

```text
原句：" 我喜欢学习。"
候选："我喜欢学习。"
```

如果自动去掉原句句首空格，就无法展示句首空格被删除。

## 5. JSON 文件上传

本版本支持上传 JSON 文件。

JSON 文件可以是：

1. 单样本 JSON；
2. 多样本 JSON 数组。

### 5.1 单样本 JSON 格式

```json
{
  "id": "sample_001",
  "source": "他对这个问题感到很有兴趣。",
  "references": [
    "他对这个问题很感兴趣。",
    "他对这个问题很有兴趣。"
  ],
  "candidates": [
    {
      "id": "model_a",
      "text": "他对这个问题很感兴趣。"
    },
    {
      "id": "model_b",
      "text": "他对这个问题非常感兴趣。"
    },
    {
      "id": "model_c",
      "text": "他对这个题目很感兴趣。"
    }
  ]
}
```

### 5.2 多样本 JSON 数组格式

```json
[
  {
    "id": "sample_001",
    "source": "他对这个问题感到很有兴趣。",
    "references": [
      "他对这个问题很感兴趣。",
      "他对这个问题很有兴趣。"
    ],
    "candidates": [
      {
        "id": "model_a",
        "text": "他对这个问题很感兴趣。"
      },
      {
        "id": "model_b",
        "text": "他对这个问题非常感兴趣。"
      }
    ]
  },
  {
    "id": "sample_002",
    "source": "我昨天去学校。",
    "references": [
      "我昨天去了学校。"
    ],
    "candidates": [
      {
        "id": "model_a",
        "text": "我昨天去了学校。"
      },
      {
        "id": "model_b",
        "text": "我昨天去过学校。"
      }
    ]
  }
]
```

### 5.3 JSON 字段要求

| 字段 | 是否必填 | 类型 | 说明 |
|---|---|---|---|
| `id` | 可选 | string | 样本编号，缺失时自动生成 |
| `source` | 必填 | 非空 string | 原句 |
| `references` | 可选 | string[] | 参考答案列表 |
| `candidates` | 必填 | array | 候选句列表 |
| `candidates[].id` | 可选 | string | 候选句编号，缺失时自动生成 |
| `candidates[].text` | 必填 | string | 候选句文本 |

如果 `references` 缺失，默认设为：

```json
"references": []
```

如果候选句 `id` 缺失，按顺序生成：

```text
candidate_1
candidate_2
candidate_3
```

### 5.4 JSON 空字符串处理规则

JSON 中不同位置的空字符串处理规则不同。

| 位置 | 处理方式 |
|---|---|
| `references` 中的空字符串 | 自动忽略 |
| `candidates[].text` 为空字符串 | 保留，表示模型输出为空 |
| `source` 为空字符串 | 报错 |
| 手动输入空参考答案 | 自动忽略 |
| 手动输入空候选句 | 自动忽略 |

例如：

```json
{
  "source": "我昨天去了学校。",
  "references": ["", "我昨天去了学校。"],
  "candidates": [
    {
      "id": "model_a",
      "text": ""
    }
  ]
}
```

处理结果为：

```text
空 reference 被忽略；
model_a 的空 text 被保留，并展示为全句删除。
```

### 5.5 重复 candidate id 处理

如果 JSON 中多个候选句使用相同 id，系统自动重命名，避免覆盖。

例如：

```json
"candidates": [
  {
    "id": "model_a",
    "text": "我昨天去了学校。"
  },
  {
    "id": "model_a",
    "text": "我昨天去过学校。"
  }
]
```

系统重命名为：

```text
model_a
model_a_2
```

如果出现第三个重复 id，则命名为：

```text
model_a_3
```

### 5.6 文件规模与按需计算规则

第一版支持中小规模 JSON 文件上传。

建议限制为：

```text
单个 JSON 文件建议不超过 10MB；
样本数量建议不超过 3000 条。
```

上传多样本 JSON 后，系统只解析并保存样本列表，不立即对所有样本批量计算 `edit_groups` 和 `render_lines`。

页面只对当前选中的样本执行：

1. 字符级对齐；
2. 编辑提取；
3. 相邻编辑合并；
4. `edit_groups` 生成；
5. `render_lines` 生成；
6. 表格渲染；
7. 多行高亮渲染；
8. JSON 预览生成。

规则总结：

```text
上传多样本 JSON 后，只对当前样本进行对齐和渲染，不进行全量批处理。
```

### 5.7 多样本切换

上传多样本 JSON 后，页面提供：

```text
当前样本：sample_001
[上一条] [下一条] [样本下拉选择]
```

切换样本时，必须清空旧样本展示，并重新生成当前样本的：

1. `targets`；
2. `edit_groups`；
3. `render_lines`；
4. 编辑组表格；
5. 多行高亮视图；
6. JSON 预览。

```text

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

```

手动输入和 JSON 上传的区别只存在于输入阶段。进入对齐模块后，两者使用完全相同的处理流程。

## 7. 文本预处理

本版本只做轻量预处理：

1. 保留原始文本；
2. 判断空输入框时可以使用 `trim()`；
3. 保存文本时保留原文，不自动 trim；
4. 过滤空输入框；
5. 保留句内空格；
6. 保留句首和句尾空格；
7. 保留原始标点；
8. 不做繁简转换；
9. 不做分词；
10. 不改写文本内容。

这样可以避免预处理改变字符位置，导致高亮错位。

## 8. 编辑提取

### 8.1 基础对齐方法

本版本使用前端字符级 diff。

实现时可采用 LCS、Myers diff 或等价字符级 diff 算法，不依赖 Python `difflib`。

字符切分时需要：

```text
使用 Array.from(text) 按字符处理，
不要直接使用 text[i] 处理字符串。
```

这样可以避免某些特殊字符、emoji 或组合字符被错误拆分。所有 `source_start`、`source_end`、`target_start`、`target_end` 均按 `Array.from` 后的字符索引计算。

基础操作包括：

| 操作 | 含义 |
|---|---|
| `equal` | 两边相同 |
| `replace` | 替换 |
| `delete` | 原句中有，目标句中没有 |
| `insert` | 原句中没有，目标句中新增 |

最终基础 edit 只输出：

```text
replace
delete
insert
```

不输出 `equal` 编辑。

### 8.2 基础编辑格式

```json
{
  "op": "replace",
  "source_start": 6,
  "source_end": 12,
  "source_text": "感到很有兴趣",
  "target_start": 6,
  "target_end": 10,
  "target_text": "很感兴趣"
}
```

字段说明：

| 字段 | 含义 |
|---|---|
| `op` | 编辑类型，`replace/delete/insert` |
| `source_start` | 编辑在原句中的起始位置 |
| `source_end` | 编辑在原句中的结束位置 |
| `source_text` | 原句中被编辑的片段 |
| `target_start` | 编辑在目标句中的起始位置 |
| `target_end` | 编辑在目标句中的结束位置 |
| `target_text` | 目标句中的修改片段 |

## 9. 编辑合并规则

字符级 diff 可能产生过碎的编辑，因此需要合并相邻小编辑。

本版本采用以下合并条件：

```text
两个编辑之间未修改字符数 ≤ 2；
合并后的 source_text 长度 ≤ 12；
合并后的 target_text 长度 ≤ 12；
两个编辑之间的未修改片段不包含明显标点。
```

只有同时满足以上条件，才将相邻编辑合并为一个展示编辑块。

明显标点包括：

```text
，。！？；：、,.!?;:
```

合并后的编辑类型规则：

| 合并前情况 | 合并后类型 |
|---|---|
| 只有 delete | delete |
| 只有 insert | insert |
| 只有 replace | replace |
| delete + insert | replace |
| delete + replace | replace |
| insert + replace | replace |
| delete + insert + replace | replace |

只要合并块中同时涉及原句片段和目标句片段，就统一视为 `replace`。

## 10. edit_groups 设计

### 10.1 edit_group 作用

`edit_group` 表示：

```text
原句中的某个片段，在参考答案和不同候选句中分别变成了什么。
```

示例：

```json
{
  "group_id": "g1",
  "source_start": 6,
  "source_end": 12,
  "source_text": "感到很有兴趣",
  "items": {
    "ref_1": {
      "text": "很感兴趣",
      "op": "replace"
    },
    "ref_2": {
      "text": "很有兴趣",
      "op": "replace"
    },
    "candidate_1": {
      "text": "很感兴趣",
      "op": "replace"
    },
    "candidate_2": {
      "text": "非常感兴趣",
      "op": "replace"
    }
  }
}
```

### 10.2 重叠 span 合并规则

多个目标句的编辑 span 可能不完全相同，但只要在原句坐标上有重叠，就归入同一个 `edit_group`。

合并后：

```text
group_start = min(edit.source_start)
group_end = max(edit.source_end)
source_text = source[group_start:group_end]
```

### 10.3 edit_group.items.text 计算规则

`edit_group.items[target_id].text` 不应简单等于某个 edit 的 `target_text`。

它表示：

```text
该 target 在整个 edit_group 对应原句范围内的最终展示文本。
```

也就是说，先确定 `edit_group` 的原句范围：

```text
group_start = min(edit.source_start)
group_end = max(edit.source_end)
source_text = source[group_start:group_end]
```

然后对每个 target，计算该 target 在这一整段范围内最终对应的文本。

例如：

```text
原句片段：感到很有兴趣
参考答案：很感兴趣
候选句：很有兴趣
```

即使候选句底层 edit 只是：

```text
感到 → 空
```

在表格中也应该显示：

```text
很有兴趣
```

而不是只显示空。

规则如下：

1. 如果该 target 在该 group 范围内没有修改，则显示原句片段，`op = equal`；
2. 如果该 target 将整个片段替换为其他文本，则显示替换后的文本，`op = replace`；
3. 如果该 target 删除了整个片段，则显示被删除内容，`op = delete`；
4. 如果该 target 只删除 group 中的一部分，则显示该 group 范围在 target 中最终剩余的文本；
5. 如果该 group 是纯插入，则原句侧显示 `/\`，插入 target 显示 `/插入内容\`，没有插入的 target 也显示 `/\`。

### 10.4 插入与相邻编辑合并

插入操作满足：

```text
source_start == source_end
```

如果插入位置距离已有 `edit_group` 边界 ≤ 1 个字符，则合并到该 `edit_group`；否则单独成组。

纯插入 `edit_group` 的原句侧不能显示为空白，应显示：

```text
/\
```

### 10.5 纯插入 edit_group 展示规则

当某个 `edit_group` 是纯插入时，`source_text` 可能为空字符串。此时表格和高亮视图不能显示为空白。

统一显示规则：

```text
原句侧：/\
插入 target：/插入内容\
未插入 target：/\
```

例如：

```text
原句：我昨天去学校。
参考：我昨天去了学校。
候选：我昨天去学校。
```

表格展示：

| 原句片段 | 参考 | 候选 |
|---|---|---|
| /\ | /了\ | /\ |

### 10.6 edit_groups 排序规则

`edit_groups` 需要按原句位置稳定排序。

排序规则：

1. 按 `source_start` 升序排序；
2. 如果 `source_start` 相同，非纯插入 group 优先；
3. 纯插入 group 按生成顺序稳定排序；
4. 句首插入显示在第一个字符前；
5. 句尾插入显示在最后一个字符后；
6. 句中插入显示在对应位置之后。

## 11. render_lines 设计

### 11.1 为什么需要 render_lines

`edit_groups` 主要用于编辑组表格展示，但仅靠 `edit_groups` 不足以稳定生成多行高亮视图。

原因是，多行高亮视图需要知道每个参考答案或候选句中，哪些文本片段对应哪个 `edit_group`。如果只根据 `edit_group.items.text` 在句子中搜索位置，遇到重复文本时容易高亮错位。

因此，本版本在 `edit_groups` 之外，增加 `render_lines` 结构。

```text
edit_groups：用于表格展示；
render_lines：用于多行句子高亮展示。
```

多行高亮视图必须使用 `render_lines` 渲染，不应通过字符串搜索 `edit_group.items.text` 来定位高亮位置。

### 11.2 render_lines 示例

`render_lines` 中每一行对应原句、某个参考答案或某个候选句。每一行由若干 `segment` 组成。

示例：

```json
{
  "render_lines": [
    {
      "id": "source",
      "type": "source",
      "label": "原句",
      "text": "他对这个问题感到很有兴趣。",
      "segments": [
        {
          "text": "他对这个",
          "type": "plain"
        },
        {
          "text": "问题",
          "type": "replace",
          "group_id": "g1"
        },
        {
          "text": "感到很有兴趣",
          "type": "replace",
          "group_id": "g2"
        },
        {
          "text": "。",
          "type": "plain"
        }
      ]
    }
  ]
}
```

### 11.3 输出数据格式

最终输出结构为：

```json
{
  "id": "sample_001",
  "source": "他对这个问题感到很有兴趣。",
  "targets": [],
  "edit_groups": [],
  "render_lines": []
}
```

其中：

| 字段 | 用途 |
|---|---|
| `targets` | 当前样本的参考答案和候选句列表 |
| `edit_groups` | 用于编辑组表格展示 |
| `render_lines` | 用于多行高亮展示 |

## 12. 前端展示

本版本提供两种展示视图：

1. 编辑组表格视图；
2. 多行句子高亮视图。

### 12.1 编辑组表格

表格每一行对应一个 `edit_group`。

示例：

| 原句片段 | 参考1 | 参考2 | candidate_1 | candidate_2 | candidate_3 |
|---|---|---|---|---|---|
| 问题 | 问题 | 问题 | 问题 | 问题 | 题目 |
| 感到很有兴趣 | 很感兴趣 | 很有兴趣 | 很感兴趣 | 非常感兴趣 | 很感兴趣 |

要求：

1. 表格只展示当前样本真实产生的 `edit_groups`；
2. 当前样本没有的片段不能出现在表格中；
3. 删除、插入说明示例不能混入主表格；
4. 切换样本后表格必须重新渲染；
5. 长文本单元格自动换行；
6. 如果 `references = []`，则 `targets` 中不包含参考答案列，表格只显示候选句列，不显示空的 `ref_1` 或 `ref_2` 列。

### 12.2 多行高亮视图

多行高亮视图使用 `render_lines` 渲染。

示例：

```text
原句：        他对这个[问题][感到很有兴趣]。
参考1：      他对这个[问题][很感兴趣]。
参考2：      他对这个[问题][很有兴趣]。
candidate_1：他对这个[问题][很感兴趣]。
candidate_2：他对这个[问题][非常感兴趣]。
candidate_3：他对这个[题目][很感兴趣]。
```

要求：

1. 同一个 `edit_group` 使用同一种边框或底色；
2. 不同 `edit_group` 使用不同颜色；
3. 鼠标悬停显示原句片段和目标片段；
4. 点击高亮片段时定位到表格中的对应行；
5. 长句自动换行；
6. 高亮片段可以自然跨行；
7. 不通过字符串搜索定位高亮位置。

---

## 13. 删除展示规则

删除内容在目标句中不存在，但必须显示为删除占位符。

删除格式：

```text
[被删除内容]
```

其中：

1. 被删除文字放在方括号内部；
2. 删除斜线覆盖在被删除文字上；
3. 删除斜线不是普通文本字符，而是视觉覆盖效果；
4. 删除标记使用红色系；
5. 删除标记在表格和多行高亮视图中保持一致；
6. 删除标记可以随长句自然换行。

示例：

```text
原句：      [好的，]我明天去学校。
candidate：[好的，]我明天去学校。
```

其中 candidate 中的 `[好的，]` 是删除占位符，不表示候选句实际包含该片段。

不应渲染成：

```text
[\]好的，
[好的，\]
```

而应渲染为：

```text
[好的，]
```

并在“好的，”文字上覆盖一条删除斜线。

---

## 14. 插入展示规则

插入内容在原句中不存在，因此需要在原句侧显示插入符号。

插入格式：

```text
原句侧：/\
目标句侧：/插入内容\
```

示例：

```text
原句：      我昨天去/\学校。
candidate：我昨天去/了\学校。
```

规则：

1. 原句侧只显示 `/\`；
2. 原句侧不显示“插入点”文字；
3. 目标句侧显示 `/插入内容\`；
4. 插入内容必须放在 `/` 和 `\` 中间；
5. 原句侧和目标句侧插入标记颜色一致；
6. 插入标记使用绿色系；
7. 插入标记在表格和多行高亮视图中保持一致；
8. 插入标记可以随长句自然换行；
9. 实现中需要注意反斜线字符的转义，避免页面中漏显示 `\`。

---

## 15. 不可见字符展示

如果编辑内容涉及空格、制表符或换行，需要可视化展示。

| 字符 | 展示形式 |
|---|---|
| 普通空格 | ␠ |
| 制表符 | ⇥ |
| 换行符 | ↵ |

例如删除一个空格时，表格显示：

| 原句片段 | candidate |
|---|---|
| ␠ | [␠] |

---

## 16. 无修改样本展示

如果当前样本所有目标句都与原句完全一致，结果区显示：

```text
该样本没有检测到修改。
```

如果某个候选句没有修改，但其他目标句有修改，则该候选句在对应 `edit_group` 中显示原句片段，`op = equal`。

## 17. 长句自动换行

所有句子展示区域都必须支持长句自动换行。

适用区域包括：

1. 多行高亮视图；
2. 编辑组表格；
3. 删除标记；
4. 插入标记；
5. JSON 预览区域。

要求：

1. 长句不能横向撑破页面；
2. 高亮片段可以自然跨行；
3. 删除和插入标记可以随文本换行；
4. 表格单元格内长文本自动换行；
5. 页面整体不应出现不必要的横向溢出。

建议样式：

```css
.sentence-line {
  white-space: pre-wrap;
  word-break: break-word;
  overflow-wrap: anywhere;
  line-height: 1.8;
}

.diff-table td,
.diff-table th {
  white-space: pre-wrap;
  word-break: break-word;
  overflow-wrap: anywhere;
  vertical-align: top;
}

.highlight-span {
  display: inline;
  padding: 2px 4px;
  border-radius: 4px;
}
```

高亮片段不建议使用 `inline-block`，否则长片段可能无法自然换行。

---

## 18. 当前样本展示一致性

主编辑组表格、多行高亮视图和 JSON 预览必须来自同一个当前样本。

规则如下：

1. 当前样本没有的编辑片段不能出现在主表格中；
2. 删除示例和插入示例不能混入当前样本主表格；
3. 切换样本后，必须清空旧样本展示；
4. 切换样本后，必须重新生成当前样本的 `targets`、`edit_groups`、`render_lines`、表格、高亮视图和 JSON 预览；
5. 如果当前样本没有删除操作，主表格中不应出现删除行；
6. 如果当前样本没有插入操作，主表格中不应出现插入行。

## 20. 本版本功能范围

本版本实现以下功能：

1. 支持前端手动输入原句；
2. 支持前端手动增加、删除参考答案输入框；
3. 支持前端手动增加、删除候选句输入框；
4. 支持参考答案为空；
5. 支持上传 JSON 文件；
6. 支持单样本 JSON；
7. 支持多样本 JSON 数组；
8. 自动为缺失样本 id 生成编号；

11. 支持多样本切换；
12. 上传多样本 JSON 后按需计算当前样本，不全量批处理；

19. 支持编辑组表格视图；

## 21. 方案总结

本版本工具的核心设计是：

```text
以原句为中心做字符级对齐，
把所有参考答案和候选句的修改映射到原句坐标，
再按重叠 span 合并成 edit_groups，
同时生成 render_lines，
最后用表格和多行高亮展示差异。
```

其中：

```text
edit_groups：用于编辑组表格；
render_lines：用于多行高亮展示。
```

本版本不依赖复杂语义判断，也不进行错纠类型判断，重点是先把原句、参考答案和候选句之间的修改差异清楚展示出来。
